下面来看一下每篇文章里必出现的名词: Skills。

我觉得在AI应用领域，出现什么名词都别太害怕，大概率都是蛇套了个马甲变王八了。除非你用的是多模态大模型，那么一般最终输入给模型的还是一大段Token罢了。围绕着限制格式或者其他trick，来唤起围绕在周边的一堆工具来工作。

如果说 Prompt 是临时的文本指令，那么 Skills 是一套结构化的、可持久化存储且包含执行逻辑的标准化能力单元。

根据2025年末至2026年初的行业标准（特别是Anthropic推出的Claude Skills及后续开源协议），Skills 在技术实现上包含三个核心组成部分：

  1. 结构化的指令集（Structured Instructions）：
    不同于自由文本的 Prompt，Skills 内部包含经过验证的、固定格式的系统指令。这些指令明确规定了任务的目标、执行步骤、约束条件以及输出格式。例如，一个“代码审查 Skill”不仅包含“请检查代码”这句话，还内置了具体的检查列表（如安全性、性能、规范性）、错误分级标准以及报告模板。这消除了每次对话中重新定义规则的不确定性。
  2. 绑定的工具与API配置（Bound Tools & API Configurations）：
    Skills 直接封装了对外部工具的调用权限和参数配置。在传统的 Tool Use 模式下，开发者需要为每个 API 单独编写函数定义（Function Definition）；而在 Skills 架构中，特定的 API 端点、认证密钥（在安全沙箱内）、输入输出映射逻辑被预置在技能包中。当用户激活该 Skill 时，模型自动获得调用这些特定接口的能力，无需在当前对话上下文中重复声明。
  3. 领域知识库与上下文预设（Domain Knowledge & Context Presets）：
    Skills 可以挂载特定的向量数据库索引或静态知识文档。这使得模型在执行任务时，能够直接访问该领域的专业术语、历史案例或企业规范，而不仅仅依赖其预训练数据。这种机制将通用的语言能力转化为特定场景的专业执行能力。

但是不管怎么包装，最终给到大模型“看”的，依然是一段文本（Token）。从最底层的原理来说，输入接口从未改变过：它只能接收序列化的文本数据。无论上层封装得多么复杂，什么文件夹、JSON配置、API调用等等，在送入模型推理引擎的前一刻，系统必须将所有信息“拍扁”成一段连续的字符流。没有对模型本身有修改比如各种微调操作。
